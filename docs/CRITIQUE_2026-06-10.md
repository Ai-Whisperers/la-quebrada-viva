# CRITIQUE_2026-06-10 — Honest Roast of the Repo

Source-of-truth analysis document. Persists the 8-section in-session critique so it survives compaction and future sessions can act on it. Authoritative for "what is wrong with this repo". Companion to `UPGRADE_PLAN.md` (the fix), `sub_render_strategy.md` (the new workflow), and `SESSION_LOG.md` tick 21 (the audit trail).

This document is **intentionally adversarial**. The user asked for a roast — no sycophancy. The repo has real strengths (18/18 finals shipped, doc mesh closed, license posture clean) and they are not relitigated here. What follows is what is broken, missing, or load-bearing-by-accident.

---

## 1. Repo hygiene

- **`scene.blend` × 3 redundancy**: `scene.blend` (13 MB) + `scene.blend1` (13 MB Blender auto-backup) + `scene.blend.session-backup` (13 MB shell-script backup) = **~39 MB of duplicated binary on a single disk**. None of them are in git (correctly gitignored), but they consume working-tree space and confuse new sessions about which is canonical. The shell scripts produce `scene.blend.session-backup`, Blender produces `scene.blend1`, and `build_scene.py` overwrites `scene.blend`. No documentation tells you that.
- **`__pycache__/` is not gitignored** at repo root. Currently empty/tracked-by-accident on some sessions. One stray commit and we ship `.pyc` files.
- **`_archive/` carries 5.6 MB of `.bak`** — `build_scene.py.pre-refactor.bak` and friends. CLAUDE.md tells future sessions "ignore it", but it's tracked in git history. It's not load-bearing; it's archaeology.
- **No git remote**. Single-disk SPOF with escritura 17 days out. The 18 finals + the entire planning corpus exist on one Linux laptop. If the SSD dies before the closing meeting, the deliverable dies with it.
- **`.env.local` ungitignored** (1524 bytes, untracked). Whether it contains secrets or just dev-shell flags, it is not protected by `.gitignore` — only by the fact that staging is explicit (`git add <file>`, never `-A`). One distracted commit using `-A` (against the standing rule) leaks it.
- **3 redundant `extract_gedi_*.py` variants** (`extract_gedi.py`, `extract_gedi_https.py`, `extract_gedi_s3.py`). All three exist because the S3 path 403'd on the cloud-pool EULA and we improvised an HTTPS fallback. None were ever pruned. Future sessions will not know which one is canonical.
- **`scripts/mcp_daemon.py` untracked-forever**. The standing rule "never stage `mcp_daemon.py`" is correct (local dev socket), but the file's permanent untracked status fights `git status --short` every session and we have to repeat the exclusion in every batch commit body.

## 2. Code in `lqv/`

- **`lqv/materials.py` = 341 lines of registry bloat**. Single global `MAT` dict, mixed concerns (image loading + principled factory + procedural variegation + per-asset wiring). One typo in a key string is a runtime KeyError on render — not a static check. No type hints on the factory.
- **14 stub files** (8 typologies in `lqv/typologies/` + 6 amenities in `lqv/amenities/`), each ~20-25 lines of dormant scaffolding. They exist because the housing-park concept proposed 8 typologies + 6 amenity zones, but none of them actually build geometry. They're commitments without execution. Either populate them or delete them; the middle state is the worst state.
- **`build_scene.py` RNG fragility (113 lines)**: the seed call must sit AFTER `materials.build_materials()` and BEFORE the first `build_*` call. ARCHITECTURE.md documents this. There is no automated test that enforces it. A new contributor reordering the file silently shifts every petal, every grass tuft, every anthurium — and nothing fails.
- **`lqv/engine.py` silent CPU fallback (72 lines)**: the GPU device cascade (OPTIX/CUDA/HIP/METAL/ONEAPI) is wrapped in `except Exception: pass`. If the GPU init fails, we silently drop to CPU with no log. The actual reason — "no ROCm installed for the AMD RX 6400" — is buried in STATUS.md but invisible at runtime.
- **Dormant util audits**: `lqv/util/ten_rules_check.py` (122 lines) and `lqv/util/material_audit.py` (108 lines) and `lqv/util/random_audit.py` (102 lines) exist as policy enforcement tools but are not wired into `smoke_test.sh` or any pre-commit hook. They are evidence of an intent to enforce; they enforce nothing.
- **No type hints, no `pyproject.toml`, no `ruff`/`mypy`**. The codebase is "scripty Python". For a one-off render it's fine; for an asset-park system meant to expand to 8 typologies + 6 amenities it is technical debt.

## 3. Docs

- **29 `.md` files in `docs/` totalling roughly 470 KB of prose**. That is more text than the entire `lqv/` Python codebase (~3738 LOC). The doc:code ratio is upside-down for a renderer.
- **`docs/SESSION_LOG.md` = 76 KB / 384 lines**. It is a narrative archaeology of every tick of every session, and it is the longest-lived doc-mesh maintenance burden in the repo. Half of it is reciprocal back-pointer extensions that exist to satisfy the doc-mesh closure invariant, not because anyone will read them.
- **4 overlapping research docs**: `paraguay_clay_house_research.md` + `MASTER_BRIEF.md` + `EUROPEAN_TOURISM_SPEC.md` + `HOUSING_PARK_CONCEPT.md` all describe the project from overlapping angles. Cross-references try to make them additive, but a new reader can't tell which is authoritative without reading all four.
- **3-copy 10-rules drift risk**: the 10 design rules appear in `MASTER_BRIEF.md` §14, in `CLAUDE.md`, and in `docs/research/README.md`. Three sources of truth = guaranteed drift. The first session that edits one and not the others will create a silent inconsistency.
- **`HOUSING_PARK_CONCEPT.md` has 25 open questions for Wesley** that have been open for the entire session. They block business decisions, not render decisions, but their presence in the doc gives a false sense that the concept is "in progress".
- **Doc-mesh maintenance > code maintenance**. The last three sessions spent more energy on back-pointer extensions (Batches 11/12) than on code or renders. This is a smell.
- **`docs/AI_WHISPERERS_STYLE.md` is listed twice** in `CLAUDE.md` (lines 16 and 17 are duplicates).

## 8. Honest roast

The repo is **over-documented as an artifact and under-engineered as a product**.

- 29 doc files. 3738 LOC of Python. The session-log is a journal, not a deliverable.
- The 18/18 finals shipped is real — but the path to them is fragile (RNG ordering, MAT registry, positional coupling, hidden Boolean cutters, all undocumented at the code level), and the petal defect is acknowledged-but-deferred under a directive that protects byte-identity over fixing the bug.
- The doc-mesh closure invariant ("every doc must be bidirectionally reachable") is intellectually pretty and operationally a tax. Two of the last three batches were doc-mesh maintenance. That energy could have produced more renders, a real survey commission, or a GitHub push.
- 17 days to escritura. **No git remote.** The single most important risk in the repo is that a hardware failure destroys the deliverable, and the fix takes 90 seconds (`git remote add origin <url> && git push -u origin master`). It has not been done.

**Strongest opinion**: stop extending the doc mesh. Push to GitHub. Generate the Wesley PDF. Fix the petals. Ship the survey commission. Everything else can wait.
